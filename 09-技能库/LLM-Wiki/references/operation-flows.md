## 一、Ingest（入库）—— 流水线操作详解

### 1.1 触发场景

| 触发词 | 示例 |
|--------|------|
| 处理 Notes/ | "处理一下 Notes/" |
| 入库新资料 | "帮我把这份文档入库" |
| 分发知识 | "把这些内容分发到知识库" |

### 1.2 标准操作流程

```
Step 1: 扫描 Notes/
┌─────────────────────────────────────────────────────────┐
│ AI 扫描 Notes/ 下所有没处理的内容                       │
│ 扫描范围：Clippings/、Inbox/、Conversation/            │
└─────────────────────────────────────────────────────────┘

Step 2: 生成方案表
┌─────────────────────────────────────────────────────────┐
│ 文件 | 建议放哪 | 怎么处理 | 为什么                     │
│------|---------|---------|-------                       │
│ a.md | Knowledge/ | 新建 | 全新主题                    │
│ b.md | Software/ClaudeCode/ | 合并 | 与现有文档重叠    │
│ c.md | Notes/ | 留着 | 内容太短，等补充                 │
└─────────────────────────────────────────────────────────┘

Step 3: 等待确认
┌─────────────────────────────────────────────────────────┐
│ "以上是处理方案，请确认后我执行分发"                     │
│ 用户说"确认"或"按这个执行"                             │
└─────────────────────────────────────────────────────────┘

Step 4: 执行分发
┌─────────────────────────────────────────────────────────┐
│ 按确认的方案移动文件到目标目录                          │
│ 创建必要的目录                                          │
└─────────────────────────────────────────────────────────┘

Step 5: 知识加工
┌─────────────────────────────────────────────────────────┐
│ 1. 读原文，提取关键信息                                │
│ 2. 写摘要（如果需要）                                   │
│ 3. 创建或更新 Wiki 页面                                │
│ 4. 补交叉引用（强关联优先）                            │
└─────────────────────────────────────────────────────────┘

Step 6: 清理 Notes/
┌─────────────────────────────────────────────────────────┐
│ 从 Notes/ 删除已分发的文件                             │
│ 更新 Notes/ 的 Inbox 状态                              │
└─────────────────────────────────────────────────────────┘
```

### 1.3 分发判断标准

| 内容类型 | 判断标准 | 目标目录 |
|---------|---------|---------|
| **软件工具** | 涉及具体工具的使用技巧 | Software/ |
| **开发技巧** | 代码、编程相关 | Software/开发技巧/ |
| **AI 产品** | AI 产品思考、评测 | Software/AI工具/ |
| **产品研发** | 产品规划、研发流程 | Software/产品研发/ |
| **方法论** | 系统性方法论、框架 | Knowledge/方法论/ |
| **读书笔记** | 书籍阅读笔记 | Knowledge/读书笔记/ |
| **原创思考** | 自己产生的洞察 | Knowledge/原创思考/ |
| **领导力** | 领导力、团队管理 | Knowledge/管理/ |
| **创业** | 创业方法论、案例 | Knowledge/创业/ |
| **个人成长** | 哲学、沟通、成长 | Knowledge/个人成长/ |
| **投资** | 投资笔记、复盘 | LifeOS/投资/ |
| **健康** | 健身、饮食、健康 | LifeOS/健康/ |
| **保险** | 保险配置 | LifeOS/保险/ |
| **人脉** | 联系人管理 | LifeOS/人脉/ |
| **内容创作方法论** | 创作方法论 | Writing/Knowledge/ |
| **视频脚本** | 视频脚本 | Writing/脚本/ |
| **文章** | 正式文章 | Writing/文章/ |
| **运营 SOP** | 运营流程 | Writing/SOP/ |
| **选题素材** | 选题、素材 | Writing/素材/ |
| **判断不了** | 不确定放哪 | 留在 Notes/ |

### 1.4 方案表示例

```
┌─────────────────────────────────────────────────────────────────────────┐
│                            Ingest 方案表                                  │
├──────────────┬──────────────────────────┬────────────┬─────────────────────┤
│ 文件         │ 建议放哪                 │ 怎么处理  │ 为什么              │
├──────────────┼──────────────────────────┼────────────┼─────────────────────┤

├──────────────┼──────────────────────────┼────────────┼─────────────────────┤

├──────────────┼──────────────────────────┼────────────┼─────────────────────┤
```

## 二、Query（查询）—— 知识调用详解

### 2.1 触发场景

| 触发词 | 示例 |
|--------|------|
| 问知识库 | "帮我查一下知识库里的..." |
| 搜索 | "搜索知识库里关于..." |
| 调用知识 | "用知识库里的内容回答..." |

### 2.2 标准操作流程

```
Step 1: 用户提问
┌─────────────────────────────────────────────────────────┐
│ 用户问："创业怎么找前 1000 个用户？"                   │
└─────────────────────────────────────────────────────────┘

Step 2: 搜索 Wiki
┌─────────────────────────────────────────────────────────┐
│ 按优先级搜索：                                          │
│ 1. 摘要页面（概括性内容）                               │
│ 2. 实体页面（具体案例）                                 │
│ 3. 概念页面（方法论）                                   │
│                                                           │
│ 搜索关键词：创业、获客、用户增长                        │
└─────────────────────────────────────────────────────────┘

Step 3: 综合回答
┌─────────────────────────────────────────────────────────┐
│ 整合多篇文件的内容，给出综合答案                        │
│ 引用来源，标注不确定性                                  │
└─────────────────────────────────────────────────────────┘

Step 4: 反哺建议（关键！）
┌─────────────────────────────────────────────────────────┐
│ 如果发现两篇文件内容互补但没互相链接：                  │
│                                                           │
│ "发现一个关联建议：《获客方法论》和《产品打磨哲学》     │
│   内容互补，但目前没有互相链接。建议补上吗？"          │
│                                                           │
│ 用户确认后，补上链接                                    │
└─────────────────────────────────────────────────────────┘
```

### 2.3 搜索优先级策略

| 优先级 | 页面类型 | 搜索重点 |
|--------|---------|---------|
| 1 | 摘要页面 | 概括性内容、一句话说明 |
| 2 | 实体页面 | 具体案例、数据、人物 |
| 3 | 概念页面 | 方法论、理论框架 |
| 4 | 综合结论 | 跨资料洞察 |

### 2.4 反哺触发条件

| 触发条件 | 示例 |
|---------|------|
| 内容互补 | A 讲获客，B 讲产品，互相补充 |
| 理论-实践配对 | 方法论文档 + 应用案例文档 |
| 读书-应用配对 | 读书笔记 + 实践反思 |
| 人物-方法配对 | 人物案例 + 对应方法论 |

## 三、Lint（健康检查）—— 维护流程详解

### 3.1 触发场景

| 触发词 | 示例 |
|--------|------|
| 健康检查 | "跑一下 Lint" |
| 知识库检查 | "检查一下知识库状态" |
| 维护 | "维护一下知识库" |

### 3.2 四项检查标准

| 检查项 | 标准 | 检测方法 | 处理方式 |
|-------|------|---------|---------|
| **孤立页面** | 没有任何链接指向 | 检查反向链接 | 报告，不自动删 |
| **过时内容** | 超过 90 天没动过 | 检查文件修改日期 | 标记待更新 |
| **缺失引用** | 讲同一个概念但没互相链接 | 语义相似度分析 | 建议补链接 |
| **Notes/ 积压** | 超过 7 天没处理的输入 | 检查文件创建日期 | 提醒处理 |

### 3.3 按目录分跑规则

```
Knowledge/ ← 重点跑
  - 50 篇管理笔记零链接 → 问题！
  - 需要密集的交叉引用

Software/ ← 简单跑
  - 工具备忘文档本来独立
  - 孤立页面是正常的

LifeOS/ ← 按需跑
  - 投资笔记需要关联
  - 健康计划独立即可

Writing/ ← 产出后跑
  - 写完文章后检查引用
  - 作为产出前的最后一步
```

### 3.4 Lint 报告格式

```
┌─────────────────────────────────────────────────────────────────────────┐
│                           Lint 健康检查报告                             │
├─────────────────────────────────────────────────────────────────────────┤
│ 检查目录：Knowledge/                                                    │
│ 检查时间：2026-04-10                                                   │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│ 📊 孤立页面（建议处理）                                                 │
│ ─────────────────────────────────────                                 │
│ 1. 战略规划_2026.md - 无任何页面链接                                    │
│ 2. 团队激励方法.md - 无任何页面链接                                    │
│                                                                          │
│ ⏰ 过时内容（90天未更新）                                               │
│ ─────────────────────────────────────                                 │
│ 1. AI趋势_2025.md - 上次修改：2025-12-01                               │
│ 2. 市场分析_2025Q4.md - 上次修改：2025-10-15                            │
│                                                                          │
│ 🔗 缺失引用（建议补链）                                                 │
│ ─────────────────────────────────────                                 │
│ 1. 《深度工作》笔记 与 时间管理 方法论 - 概念相关                       │
│ 2. 乔布斯管理课 与 领导力框架 - 人物与方法配对                         │
│                                                                          │
│ 📥 Notes/ 积压（7天未处理）                                             │
│ ─────────────────────────────────────                                 │
│ 1. 碎片想法_04.md - 创建于：2026-04-03                                 │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘

请确认处理方式：
- [ ] 删除孤立页面
- [ ] 更新过时内容
- [ ] 补上建议的链接
- [ ] 处理 Notes/ 积压
```

## 四、确认机制详解

### 4.1 为什么需要确认

> AI 对"这篇文章应该放哪"的判断经常不靠谱。

**典型问题**：
1. 内容创作方法论，AI 放到了 Knowledge/ 而不是 Writing/Knowledge/
2. Chrome DevTools 教程，AI 新建了一个文件，其实应该合并到已有文档

### 4.2 确认流程

```
┌─────────────────────────────────────────────────────────┐
│ AI 生成方案                                              │
│   - 列出分发建议                                         │
│   - 说明判断理由                                         │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│ 用户确认                                                │
│   - 看一眼表格（成本很低）                               │
│   - 同意 / 修改 / 补充                                  │
└─────────────────────────────────────────────────────────┘
                         ↓
┌─────────────────────────────────────────────────────────┐
│ AI 执行                                                  │
│   - 按确认的方案执行                                     │
│   - 执行后报告                                          │
└─────────────────────────────────────────────────────────┘
```

### 4.3 分类的本质

> **分类这件事，本质上反映的是你自己的思维结构。AI 可以建议，但最终得你来定。**

## 五、对话沉淀详解

### 5.1 Notes/Conversation/ 的价值

> 你跟 AI 认真讨论一个问题的时候，对话过程本身就在生产知识。比如这篇文章的雏形，就是我跟 Claude Code 讨论 Karpathy 方案时聊出来的。把它丢掉太可惜了。

### 5.2 什么对话值得沉淀

| 值得沉淀 | 不值得沉淀 |
|---------|-----------|
| 深度讨论一个问题 | 简单问答 |
| 产生了新的洞察 | 事实查询 |
| 方案设计讨论 | 任务执行确认 |
| 理论推导过程 | 日常闲聊 |

### 5.3 对话沉淀格式

```markdown
# 对话沉淀：Karpathy LLM Wiki 讨论

**日期**：2026-04-10
**参与者**：悟空 & 龙龟神将
**主题**：Karpathy LLM Wiki 方案讨论

## 核心洞察

1. 知识是弹药库，产出才是开枪
2. AI 建议，人类拍板
3. 对话本身也是知识

## 讨论要点

- Karpathy 的三层架构
- 悟空的五层改造
- 确认机制的必要性
- 链接策略的选择

## 产出

- 升级 LLM Wiki Skill v2.0
- 新增操作流程文档
```

## 六、index.md（索引）详解

### 6.1 索引原则

**Karpathy**：index.md 到文件级
**悟空的实践**：只索引到目录

> 文件级索引是维护噩梦——每加一篇文章就得更新，目录结构稳定后基本不用动。
