# 七步问题解决法

## 一、核心定义

**七步问题解决法**（Seven-Step Problem Solving）是麦肯锡咨询方法论的核心框架，通过七个结构化步骤，将模糊问题转化为清晰、可执行的解决方案。

**七步框架**：

| 步骤 | 名称 | 核心问题 |
|------|------|---------|
| 第一步 | 定义问题 | 真正的问题是什么？ |
| 第二步 | 分解问题 | 问题可以分解为哪些子问题？ |
| 第三步 | 优先排序 | 哪些问题最重要？ |
| 第四步 | 制定计划 | 如何分析这些问题？ |
| 第五步 | 分析问题 | 关键假设是什么？如何验证？ |
| 第六步 | 综合建议 | 分析结论是什么？建议是什么？ |
| 第七步 | 汇报执行 | 如何呈现并推动执行？ |

**核心洞察**：问题解决的质量，取决于问题定义的质量。

---

## 二、第一步：定义问题

### 2.1 问题定义的重要性

**"问题定义错，一切皆错"**

| 错误定义 | 正确示范 |
|---------|---------|
| "销售额下降" | "2025年Q2销售额同比下降15%，主要原因是华北区域KA客户流失" |
| "员工不积极" | "核心销售团队近3个月离职率上升20%，且在职员工满意度下降至65分" |
| "竞争压力大" | "竞争对手A在华东市场份额增长30%，其核心优势是价格低20%和服务响应快" |

### 2.2 问题定义工具：问题陈述模板

**问题陈述 = 问题 + 背景 + 影响 + 边界**

```markdown
## 问题陈述模板

### 问题（Problem）
[具体描述问题是什么]

### 背景（Context）
[问题发生的背景和前提条件]

### 影响（Impact）
[问题造成的影响和损失]

### 边界（Scope）
[问题的范围和边界]
```

**问题陈述示例**：

```
问题：电商平台用户转化率从8%下降至5%
背景：近6个月完成APP改版，同时竞品密集发布新功能
影响：月GMV减少2000万，新客获取成本上升35%
边界：仅涉及移动端，PC端转化率稳定
```

### 2.3 问题定义检查

| 检查项 | 问题 |
|--------|------|
| 具体性 | 问题是否具体可测量？ |
| 可解决性 | 问题在可控范围内吗？ |
| 相关性 | 解决问题有价值吗？ |
| 时效性 | 问题现在需要解决吗？ |

## 三、第二步：分解问题

### 3.1 问题分解原则：MECE

**MECE = 相互独立（Mutually Exclusive）+ 完全穷尽（Collectively Exhaustive）**

| 分解方式 | 分解示例 |
|---------|---------|
| **按流程分解** | 漏斗转化：曝光→点击→注册→下单→支付 |
| **按结构分解** | 组织架构：总部→区域→城市→门店 |
| **按要素分解** | 营收=客单价×客流量×转化率 |
| **按原因分解** | 内因+外因 / 近因+远因 |

### 3.2 问题分解工具

| 工具 | 适用场景 |
|------|---------|
| **逻辑树** | 层层分解，穷尽所有可能性 |
| **公式分解** | 将结果分解为可计算的因子 |
| **流程分解** | 按业务环节分解 |
| **矩阵分解** | 按两个维度交叉分解 |

### 3.3 问题分解示例

**问题**：如何提升用户留存率？

```
如何提升用户留存率？
├── 产品层面
│   ├── 核心功能体验
│   ├── 新手引导
│   └── 产品稳定性
├── 运营层面
│   ├── 用户激活策略
│   ├── 留存激励体系
│   └── 内容运营
├── 渠道层面
│   ├── 渠道质量
│   ├── 用户匹配度
│   └── 归因准确性
└── 市场层面
    ├── 竞争压力
    ├── 用户需求变化
    └── 外部环境
```

## 四、第三步：优先排序

### 4.1 优先排序原则：二八法则

**80%的价值来自20%的问题**

| 优先级 | 标准 | 说明 |
|--------|------|------|
| **P0-关键** | 高影响×高可行 | 立即行动 |
| **P1-重要** | 高影响×低可行 | 规划行动 |
| **P2-一般** | 低影响×高可行 | 尽快完成 |
| **P3-观察** | 低影响×低可行 | 暂缓关注 |

### 4.2 优先排序工具

| 工具 | 说明 |
|------|------|
| **影响-可行性矩阵** | 四象限评估优先级 |
| **ICE评分** | Impact×Confidence×Ease |
| **RICE评分** | Reach×Impact×Confidence×Effort |
| **MoSCoW** | Must/Should/Could/Won't |

### 4.3 优先排序示例

| 子问题 | 影响(1-5) | 可行(1-5) | 综合(乘积) | 优先级 |
|--------|----------|-----------|-----------|--------|
| 核心功能体验差 | 5 | 4 | 20 | P0 |
| 新手引导缺失 | 4 | 5 | 20 | P0 |
| 渠道质量低 | 3 | 3 | 9 | P2 |
| 内容运营弱 | 4 | 2 | 8 | P1 |

## 五、第四步：制定分析计划

### 5.1 分析计划内容

| 内容 | 说明 |
|------|------|
| **分析目标** | 每个子问题要回答什么问题 |
| **分析方法** | 如何进行分析（数据/访谈/案头） |
| **数据来源** | 需要什么数据，从哪里获取 |
| **时间规划** | 每个分析的时间节点 |
| **资源需求** | 需要什么人力和资源 |

### 5.2 分析计划模板

```markdown
## 分析计划

| 子问题 | 分析目标 | 方法 | 数据来源 | 时间 | 负责人 |
|--------|---------|------|---------|------|--------|
| [子问题1] | [要回答的问题] | [方法] | [数据] | [日期] | [X] |
| [子问题2] | [要回答的问题] | [方法] | [数据] | [日期] | [X] |
```

### 5.3 分析计划检查

| 检查项 | 标准 |
|--------|------|
| **假设明确** | 关键假设是否清晰？ |
| **方法可行** | 分析方法是否可行？ |
| **数据可得** | 所需数据是否可以获取？ |
| **时间合理** | 时间节点是否合理？ |
| **资源匹配** | 资源是否足够？ |

## 六、第五步：分析问题

### 6.1 分析执行步骤

```
第一步：验证假设
├── 提出假设
├── 收集数据验证
└── 确认/推翻假设

第二步：数据分析
├── 描述性分析（是什么）
├── 诊断性分析（为什么）
└── 预测性分析（会怎样）

第三步：形成洞察
├── 归纳关键发现
├── 提炼核心洞察
└── 验证洞察可靠性
```

### 6.2 分析方法选择

| 问题类型 | 推荐方法 |
|---------|---------|
| **归因问题** | 相关分析、回归分析 |
| **对比问题** | 分组对比、趋势分析 |
| **构成问题** | 漏斗分析、占比分析 |
| **预测问题** | 情景分析、模型预测 |

### 6.3 分析质量检查

| 检查项 | 标准 |
|--------|------|
| **数据可靠** | 数据来源是否可靠？样本是否足够？ |
| **方法正确** | 分析方法是否适合问题？ |
| **逻辑严谨** | 推理过程是否有漏洞？ |
| **结论支持** | 结论是否有充分证据支持？ |

---

## 七、第六步：综合建议

### 7.1 结论形成步骤

```
第一步：汇总发现
├── 各子问题分析结论
├── 关键数据支撑
└── 核心洞察提炼

第二步：优先级排序
├── 按重要性排序
├── 按可行性排序
└── 综合排序

第三步：形成建议
├── 战略建议（方向）
├── 战术建议（具体行动）
└── 实施建议（路径和资源）
```

### 7.2 建议结构：金字塔结构

```
┌─────────────────────────────────┐
│ 核心建议（1句话）              │
├─────────────────────────────────┤
│ 建议1：[支撑论据]              │
│   ├── 论点1.1                   │
│   └── 论点1.2                   │
├─────────────────────────────────┤
│ 建议2：[支撑论据]              │
│   ├── 论点2.1                   │
│   └── 论点2.2                   │
├─────────────────────────────────┤
│ 建议3：[支撑论据]              │
│   ├── 论点3.1                   │
│   └── 论点3.2                   │
└─────────────────────────────────┘
```

### 7.3 建议质量标准

| 标准 | 说明 |
|------|------|
| **具体性** | 建议是否具体可执行？ |
| **可行性** | 建议在资源范围内可行吗？ |
| **优先级** | 建议的优先顺序是否清晰？ |
| **风险可控** | 建议的风险是否可接受？ |

---

## 八、第七步：汇报执行

### 8.1 汇报结构

| 部分 | 内容 | 时间占比 |
|------|------|---------|
| **结论先行** | 核心建议一句话 | 1分钟 |
| **问题定义** | 问题的背景和定义 | 2分钟 |
| **分析框架** | 问题分解和分析路径 | 2分钟 |
| **关键发现** | 核心数据和洞察 | 10分钟 |
| **建议方案** | 具体建议和实施路径 | 8分钟 |
| **讨论互动** | Q&A和讨论 | 10分钟 |

### 8.2 汇报注意事项

| 要点 | 说明 |
|------|------|
| **结论先行** | 先说结论，再说理由 |
| **数据支撑** | 每个结论有数据支撑 |
| **逻辑清晰** | 遵循金字塔结构 |
| **可视化** | 复杂内容用图表呈现 |
| **互动设计** | 预留讨论时间 |

### 8.3 执行推动

| 步骤 | 内容 |
|------|------|
| **行动清单** | 明确的待办事项 |
| **责任到人** | 每个行动有负责人 |
| **时间节点** | 每个行动有截止日期 |
| **跟踪机制** | 建立定期跟踪机制 |

## 十、五色光思维整合

### 10.1 白光分析（事实·数据）

| 分析维度 | 核心问题 |
|---------|---------|
| 问题定义 | 问题是否清晰具体？ |
| 数据支撑 | 建议是否有数据支撑？ |
| 风险评估 | 风险是否充分识别？ |

### 10.2 红光直觉（感受·直觉）

| 直觉维度 | 核心问题 |
|---------|---------|
| 问题直觉 | 直觉上问题的根源是什么？ |
| 方案直觉 | 直觉上哪个方案最可行？ |
| 风险直觉 | 直觉上最大的风险是什么？ |

### 10.3 黄光价值（利益·价值）

| 价值维度 | 核心问题 |
|---------|---------|
| 客户价值 | 方案为客户创造什么价值？ |
| 商业价值 | 方案的商业回报是什么？ |
| 战略价值 | 方案的战略意义是什么？ |

### 10.4 绿光创新（创新·突破）

| 创新维度 | 核心问题 |
|---------|---------|
| 问题重构 | 能否从新角度看问题？ |
| 方案创新 | 是否有创新的解决方案？ |
| 整合创新 | 能否整合多个方案的优势？ |

### 10.5 蓝光风险（风险·挑战）

| 风险维度 | 核心问题 |
|---------|---------|
| 执行风险 | 方案执行的最大障碍是什么？ |
| 替代风险 | 是否有更好的替代方案？ |
| 遗漏检查 | 是否有重要的风险被忽略？ |
