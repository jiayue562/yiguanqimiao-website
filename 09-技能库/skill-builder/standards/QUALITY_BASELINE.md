# Skill 质量底线体系 v1.0

> **核心定位**：不是追求100分，而是确保所有Skill都达到**统一的及格线**。及格线由11个必选项与7个增强项组成，通过结构化清单确保质量可控。

---

## 📋 第一部分：质量底线检查清单（MUST HAVE）

### 一、触发机制完整性（权重：20%）

#### ✅ 必选项 1：触发关键词权重表

**及格标准**：
- [ ] 至少包含3个P0直接触发词（权重5），例如：
  - 直接动词：构建、创建、搭建、启动
  - 系统词汇：系统名称、Skill名、完整概念词
  
- [ ] 至少包含5个P1场景触发词（权重4），例如：
  - 问题场景词：困难、困惑、无法、不知道
  - 诉求词：帮我、给我、怎么办、有没有
  - 工作词：分析、学习、优化、重构
  
- [ ] 至少包含3个P2行为信号词（权重3），例如：
  - 多步骤流程信号：步骤、阶段、过程、流程
  - 标准化信号：标准化、系统化、自动化、规范化
  - 高效需求信号：重复、效率、快速、简化

- [ ] 权重计算公式清晰（P0×5 + P1×4 + P2×3，阈值≥3分）

**快速检查**：
```bash
触发词表完整性 = 
  (P0词≥3 + P1词≥5 + P2词≥3) ? "及格" : "不及格"
```

**不及格案例**：
```
❌ 只有5个通用关键词，没有权重区分
❌ 没有P2行为信号的定义
❌ 关键词列表但无应用场景说明
```

**及格案例**：
```
✅ P0（权重5）：象思维、象感知、0→1、原创
✅ P1（权重4）：困境分析、创意突破、新想法
✅ P2（权重3）：多维度思考、跨域类比、视角切换
✅ 触发阈值：任何P0 = 立即激活；累计≥3分 = 激活
```

#### ✅ 必选项 2：场景识别矩阵

**及格标准**：
- [ ] 至少覆盖3个明确的场景（S0~S9范围内）
- [ ] 每个场景都有**明确的触发信号描述**，不能笼统
- [ ] 每个场景都有**对应的引擎路由决策**（调用哪些Skills）

**快速检查**：
```
场景识别矩阵完整性 = 
  (场景数≥3 && 每个场景有信号描述 && 每个场景有路由) ? "及格" : "不及格"
```

**不及格案例**：
```
❌ 只写"需要使用该Skill"，没有具体场景
❌ 场景不超过2个
❌ 有场景但没说调用哪些其他Skills
```

**及格案例**：
```
✅ S2场景（深度理解）：用户提供文章需要深度学习 → 调用知识学习十项指令 + 五色光思维
✅ S3场景（创意突破）：需要0→1创新 → 调用象思维 + 五色光绿光
✅ S5场景（重大决策）：复杂问题分析 → 调用五色光完整序列 + 人机协同
```

### 二、核心定义与功能完整性（权重：20%）

#### ✅ 必选项 3：清晰的一句话定义

**及格标准**：
- [ ] 用一句话明确定义该Skill解决什么问题（<=50字）
- [ ] 该定义应该让完全不了解的人也能快速理解

**快速检查**：
```
定义清晰度 = 
  (字数≤50 && 包含问题+解决方案 && 无技术术语堆砌) ? "及格" : "不及格"
```

**不及格案例**：
```
❌ "这是一个用于多维度分析的智能引擎，基于五行理论构建，融合了象思维与西方模型..."
❌ "提供专业的Skill构建工具"（太笼统）
```

**及格案例**：
```
✅ 象思维Skill：教会AI从无到有的直觉洞察，激发0→1的原创突破
✅ 五色光思维：用五种思维模式同时分析同一个问题，达成全面认知和决策共识
✅ 知识学习Skill：用十项认知指令对任何文章进行系统深度学习和知识重构
```

#### ✅ 必选项 4：完整的操作步骤（3步以上）

**及格标准**：
- [ ] 清晰列出该Skill的操作步骤（至少3步，最多7步）
- [ ] 每一步都可以独立执行，不能是笼统的抽象
- [ ] 最好包含"判断标准"——如何判断这一步完成了

**快速检查**：
```
操作步骤完整性 = 
  (步骤数在3-7之间 && 每步都是具体操作 && 包含判断标准) ? "及格" : "不及格"
```

**不及格案例**：
```
❌ Step1: 理解问题
❌ Step2: 分析问题
❌ Step3: 解决问题
（太笼统，无法执行）
```

**及格案例**：
```
✅ 五色光思维：
  Step1: 进入白光模式 → 客观列出所有事实（判断标准：没有任何价值判断，只是陈述）
  Step2: 进入红光模式 → 暴露所有直觉和情感（判断标准：感受到了什么情绪？）
  Step3: 进入绿光模式 → 发散所有可能的解决方案（判断标准：至少列出10个选项）
  Step4: 进入黄光模式 → 评估每个方案的收益（判断标准：有定量的价值评分）
  Step5: 进入蓝光模式 → 识别所有风险点（判断标准：找到了致命的风险点）
  Step6: 主持人综合 → 汇总决策（判断标准：得出一个清晰的建议）
```

### 三、测试与验证完整性（权重：15%）

#### ✅ 必选项 5：至少3个测试用例

**及格标准**：
- [ ] 包含至少3个不同场景的测试用例
- [ ] 每个用例都有**清晰的输入、预期输出、实际输出**
- [ ] 测试结果明确：通过✅ / 不通过❌ / 部分通过⚠️

**快速检查**：
```
测试用例完整性 = 
  (用例数≥3 && 每个用例有输入输出 && 有检验结果) ? "及格" : "不及格"
```

**不及格案例**：
```
❌ 只有1个或2个用例
❌ 用例没有清晰的输入-输出对应
❌ 用例只是描述，没有实际测试结果
```

**及格案例**：
```
✅ 测试用例1：
   场景：用户想要创意突破方案
   输入：某企业的产品创新困局（描述）
   预期输出：3个突破性创意方案
   实际输出：✅ 生成了5个创意，其中3个符合预期
   
✅ 测试用例2：
   场景：用户想要风险评估
   输入：某项目方案（具体细节）
   预期输出：列出所有P0和P1级风险
   实际输出：✅ 识别了7个风险，其中3个用户未曾想到
   
✅ 测试用例3：
   场景：用户混乱的分析思路
   输入：用户的混乱表述
   预期输出：澄清后的清晰思路
   实际输出：⚠️ 澄清了主要问题，但还有2个细节需要再讨论
```

## 📝 第三部分：及格线细则

### 及格线的定义（96分）

**必选11项必须全部满足**，且**单项不低于7分**（保证基础完整性）。

**禁止项**（一旦出现，直接不及格）：
- ❌ 触发关键词少于3个P0或5个P1
- ❌ 没有任何测试用例
- ❌ SKILL.md少于1500字
- ❌ 没有与其他Skills的关系说明
- ❌ 没有版本迭代计划

### 优秀线的定义（150分）

- ✅ 必选11项全部≥15分（高于基础标准）
- ✅ 增强项完成≥4项
- ✅ 包含标准案例和最佳实践文档
- ✅ 已在实际项目中验证过效果

#### 阶段四：测试验证中的质量检查

```yaml
质量检查清单:
  step1_触发机制验证:

    标准: P0≥3 + P1≥5 + 场景≥3
    判断: 满足则继续，不满足则回到阶段二重新设计
  
  step2_功能完整性验证:
    检查项: 一句话定义 + 操作步骤 + 文件结构
    标准: 定义清晰 + 步骤3-7 + SKILL.md≥2000字
    判断: 满足则继续，不满足则补充内容
  
  step3_质量评分:
    方法: 按上面的11项必选逐一检查
    通过线: ≥96分
    不通过: 标注需要改进的项目，返回相应阶段
  
  step4_最终发布决策:
    >=150分: 优秀版本，直接标准化发布
    96-149分: 测试版本，建立迭代计划后发布
    <96分: 禁止发布，需要补充完善
```

## 📌 使用指南

### 对于Skill构建者（你）

**创建新Skill时**：
1. 在SKILL.md中创建一个"质量检查表"部分
2. 逐一填写11项必选的检查框
3. 计算总分
4. 若≥96分，可以发布；若<96分，按建议补充

### 对于Skill使用者（生态伙伴）

**选择使用某个Skill时**：
- 🟢 A级（≥150分）：生产环境直接使用
- 🟡 B级（96~149分）：可用，但要关注迭代计划
- 🟠 C级（60~95分）：不推荐使用，等待补完

### 对于龙心OS系统

**自动化管理**：
- 每个Skill的评分写入触发规则的"质量分数"字段
- 若评分<96分，系统自动标记为"Beta版"
- 月度自动审计：检查所有已发布Skill的评分是否还≥96分
