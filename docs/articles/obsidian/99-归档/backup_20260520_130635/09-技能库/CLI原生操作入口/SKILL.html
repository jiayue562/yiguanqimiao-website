CLI 原生操作入口 Skill

核心定位：CLI 是个人 AI OS 中人机共生的原生操作入口，锚定"人控AI 行"的交互底层。

一、核心定义

什么是 CLI（命令行界面）？

CLI（Command Line Interface）是个人 AI OS 中，人向 AI 传递指令、AI 操作全系统软硬件的标准化原生入口：

  • 对人而言：极简、高效的"指令输出端"（无需复杂操作，直接传递创意/决策）
  • 对 AI 而言：无歧义、可精准执行的"操作输入端"（替代 GUI 的视觉依赖，适配 Agent 的自动化执行特性）
  • 核心价值：人机共生的交互基础协议

技术定位

  • 层级：四大技术栈中的第一层（入口层）
  • 功能：指令解析、人格化适配、全系统调用
  • 特性：轻量化、标准化、人格化

二、核心能力

能力1：个人专属 CLI 指令体系

核心目标：围绕个人的使用习惯、核心需求设计轻量化 CLI 指令。

指令分类：

核心决策指令（人发出）

# 内容创作类
创作AI行业文章 --基于 GEO思维模型 --贴合我人格

# 工作执行类
运行数据处理脚本 --项目 xxx --优先级 高

# 知识管理类
检索我的信仰相关笔记 --时间段 本月

执行操作指令（AI 自主调用）

# 知识库操作
检索知识库 --关键词 五行人格

# 系统操作
运行每日总结 --包含 今日完成项、明日计划

# 技能调用
调用 象思维 --场景 创新突破

设计原则：

  • ✅ 极简设计：指令长度控制在 5-10 个词
  • ✅ 语义清晰：指令意图明确，无需二次确认
  • ✅ 分类体系：按功能域（创作/执行/管理）组织

能力2：全系统 CLI 化适配

核心目标：让个人 AI OS 中的所有工具、组件、外部服务都提供 CLI 接口，AI 可通过纯文本指令一键调用。

适配范围：

  • 本地文档操作
  • 云端知识库（Obsidian、IMA）
  • 办公软件集成
  • 外部 API 调用
  • Skills 执行

实现方式：

# CLI 指令到操作的映射
指令: "检索 Obsidian 笔记"
操作:
  - tool: obsidian-cli
  - action: search
  - params: {关键词: "{arg}"}

指令: "创建 IMA 笔记"
操作:
  - tool: ima-api
  - action: create_note
  - params: {内容: "{content}"}

特性：

  • ✅ 一键调用：无需人介入，AI 可直接执行
  • ✅ 状态反馈：执行过程中实时返回进度
  • ✅ 错误处理：执行失败时自动重试或降级方案

能力3：CLI 指令的「人格化适配」

核心目标：支持指令带个人语气/人格特质，让执行结果贴合个人特质。

人格化参数体系：

思维模型参数

# 指令示例
分析问题 --基于 第一性原理 --深度 3级
# 参数解析
思维模型: 第一性原理
分析深度: 深度（3级）

人格特质参数

# 指令示例
生成文案 --人格 火行人 --语气 热情感染
# 参数解析
人格类型: 火行人
语气特征: 热情感染

文化/信仰参数

# 指令示例
分析战略 --文化 心文化 --信仰 大圆满
# 参数解析
文化背景: 心文化体系
信仰根基: 大圆满见地

AI 解析流程：

  1. 提取指令中的人格化参数
  2. 同步加载对应的个人数据
  3. 调用 Skills 时传递参数
  4. 执行结果自动应用参数特征

三、核心价值

与共生逻辑的结合

「共生」实现：

  • 人少操作：CLI 实现极简指令，人无需复杂操作
  • AI 多干活：AI 自动拆解指令为底层 CLI 操作，全链路执行
  • 分工明确：人聚焦创意/决策，AI 聚焦执行

「个性化」实现：

  • 指令前缀参数化：如"按第一性原理""按长期主义视角"
  • AI 解析时同步加载，所有执行基于核心认知
  • 文化/信仰检索标签：执行时快速调用相关背景知识

四、实操指南

场景1：日常内容创作

用户指令：

创作 AI 行业文章 --主题 AI OS 发展 --基于 教员方法论

AI 执行流程：

  1. 解析指令：主题="AI OS 发展"，思维模型="教员方法论"
  2. 调用知识库 Skill：检索相关资料
  3. 调用五色光 Skill：生成结构化分析
  4. 调用文案 Skill：生成符合人格的文章
  5. CLI 一键执行全流程，人等待结果

场景2：知识库深度检索

用户指令：

检索我的笔记 --关键词 木火共生关系 --关联 象思维

AI 执行流程：

  1. 解析指令：关键词="木火共生关系"，关联="象思维"
  2. 调用 Obsidian CLI：双向链接检索
  3. 调用知识图谱 Skill：可视化关系网络
  4. 返回结构化结果（包含笔记路径、关联节点、核心洞察）

场景3：系统自动化任务

用户指令：

运行每日自动化 --包括 知识备份、日志归档、健康检查

AI 执行流程：

  1. 解析指令：任务类型="每日自动化"
  2. 拆解为子任务：
    • 备份 Obsidian 到云端
    • 归档 WorkBuddy 日志
    • 检查系统健康状态
  3. 通过 CLI 调用对应工具
  4. 实时反馈执行进度
  5. 生成执行报告

五、与四大技术栈的协同

与 Skills 层的协同

  • 调用 Skills：CLI 指令 → AI 解析 → 匹配对应 Skills
  • 传递参数：CLI 指令中的人格化参数 → 传递给 Skills
  • 接收结果：Skills 执行结果 → CLI 统一格式化反馈给人

与 MCP 层的协同

  • 协议标准化：CLI 指令的格式 → MCP 协议标准
  • 无缝调用：CLI 通过 MCP → 调用任意 Skills
  • 状态同步：CLI 执行状态 → MCP 同步给 Framework

与 Framework 层的协同

  • 任务调度：Framework 调度模块 → CLI 指令队列
  • 推理辅助：Framework 推理模块 → CLI 指令智能补全
  • 结果整合：CLI 执行结果 → Framework 整合模块

六、技术实现要点

CLI 接口设计

接口规范:
  输入格式: "命令 [参数=值] [参数=值]..."
  输出格式: "进度 | 结果 | 错误"
  状态码:
    - 0: 成功
    - 1: 部分成功
    - 2: 执行中
    - 3: 失败

错误处理

  • 指令不明确：AI 自动追问澄清
  • 工具不可用：降级到备用方案
  • 执行失败：自动重试 3 次
  • 权限不足：提示用户确认

性能优化

  • 指令缓存：常用指令预加载
  • 并行执行：独立任务并行处理
  • 增量反馈：实时推送执行进度
