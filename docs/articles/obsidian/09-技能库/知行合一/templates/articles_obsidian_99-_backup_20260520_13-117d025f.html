🔄 工作流概览

输入收集 → 表示空间 → 压缩阶段 → 泛化阶段 → 输出呈现 → 持续迭代
   ↓          ↓         ↓         ↓         ↓         ↓         ↓
目标明确   标签化拆解  核心提炼   场景落地   结构化文档  反馈优化

📋 阶段一：表示空间（标签化拆解）

Step 1.1：目标澄清

动作：明确转化目标的核心诉求 提问：

  • 这个概念/知识的核心问题是什么？
  • 转化的最终目的是什么？
  • 成功的标准是什么？

输出：

核心问题：__ 转化目的：__ 成功标准：__

Step 1.2：概念特征提取

动作：从原始概念中提取关键特征 方法：阅读原始材料，标记关键概念、定义、原理、方法

提取框架： | 特征类型 | 特征内容 | 来源证据 | |---------|---------|---------| | 核心定义 | | 原文第X段 | | 关键原理 | | 原文第X段 | | 主要方法 | | 原文第X段 | | 关系网络 | | 原文第X段 | | 应用场景 | | 原文第X段 |

Step 1.3：标签化设计

动作：设计标签体系，确保MECE和Pareto原则 设计准则：

  • Mutually Exclusive（相互独立）：标签之间不重复
  • Collectively Exhaustive（完全穷尽）：覆盖所有关键特征
  • Pareto（80/20法则）：20%标签覆盖80%价值

标签体系： | 标签层级 | 标签名称 | 标签定义 | 权重 | |---------|---------|---------|------| | L1（核心层）| | | 5 | | L1（核心层）| | | 5 | | L2（支撑层）| | | 3 | | L2（支撑层）| | | 3 | | L3（细节层）| | | 1 | | L3（细节层）| | | 1 |

Step 1.4：MECE验证

动作：验证标签体系的完整性 验证问题：

  1. 独立性检查：任意两个标签是否重叠？
    • 如有重叠，说明：__ → 调整：__
  2. 穷尽性检查：是否遗漏了关键特征？
    • 如有遗漏，说明：__ → 补充：__
  3. Pareto检查：核心标签是否覆盖80%价值？
    • 当前覆盖率：__% → 调整：__

验证结果：

  • [ ] MECE验证通过
  • [ ] Pareto原则达成（核心标签覆盖≥80%价值）
  • [ ] 标签体系确定

🌍 阶段三：泛化（场景化落地）

Step 3.1：场景识别

动作：识别核心洞察可应用的具体场景 识别方法：头脑风暴 + 场景分类

场景分类： | 场景类型 | 场景数量 | 具体场景列表 | |---------|---------|-------------| | 个人成长 | | 1. 2. 3. | | 工作实践 | | 1. 2. 3. | | 团队管理 | | 1. 2. 3. | | 学习教育 | | 1. 2. 3. | | 其他：____ | | 1. 2. 3. |

Step 3.2：优先级排序

动作：按适用性和可行性对场景排序 排序标准：

  1. 适用性（高/中/低）：核心洞察与场景的匹配度
  2. 可行性（高/中/低）：实施难度和资源需求
  3. 价值度（高/中/低）：预期带来的价值

场景评估： | 序号 | 场景描述 | 适用性 | 可行性 | 价值度 | 综合评分（1-9） | 排序 | |------|---------|-------|-------|-------|----------------|------| | 1 | | | | | | | | 2 | | | | | | | | 3 | | | | | | | | 4 | | | | | | | | 5 | | | | | | |

综合评分计算：

综合评分 = 适用性×3 + 可行性×2 + 价值度×4

📊 输出呈现

Step 4.1：结构化文档生成

动作：按照输出模板生成结构化文档 文档结构：

  1. 基本信息（目标、来源、场景）
  2. 三阶段转化结果（标签、洞察、泛化）
  3. 质量评估（理验性、实操性、创新性）
  4. 知识图谱链接（关联概念、存储位置）
  5. 持续迭代机制（验证计划、优化建议）

🔄 持续迭代

Step 5.1：验证计划

动作：设计验证方法和时间表

验证方法：

  • [ ] 实操验证：在真实场景中应用并观察效果
  • [ ] 用户反馈：收集目标用户的使用反馈
  • [ ] 专家评审：邀请领域专家评估理论框架
  • [ ] 理推演：通过逻辑推演验证内部一致性

验证时间表： | 验证类型 | 计划时间 | 责任人 | 成功标准 | |---------|---------|-------|---------| | 初次验证 | YYYY-MM-DD | | | | 中期评估 | YYYY-MM-DD | | | | 长期跟踪 | YYYY-MM-DD | | |

Step 5.2：优化机制

动作：建立持续优化流程

优化触发条件：

  • 验证反馈显示理论清晰度不足
  • 实操反馈显示实操可行性低
  • 用户反馈显示创新价值不明确
  • 发现新的应用场景或关联理论
