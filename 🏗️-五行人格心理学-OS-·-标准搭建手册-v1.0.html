③ 命名统一规范表

概念 统一名称 禁止混用
快慢思考调度引擎 凤心OS ~~总智能体~~、~~L4~~
五行分析智能体 五行分智能体（木/火/土/金/水） 不单独命名
知识地基层 L7根系 ~~知识库层~~
应用入口层 L1接待台 ~~元层次~~

④ 龙心OS 与凤心OS 边界图

龙心OS（通用认知操作系统）
├── 象思维·知识学习·五色光·人机协同·知行合一
└── 可调用 → 凤心OS（作为第3层专业智能体）

凤心OS（五行人格专项智能体）
├── 快思考（总调度）+ 慢思考（五行分智能体）
└── 可调用 → 龙心OS七模块（协同时）

调用规则：
  - 龙心OS处理通用认知任务
  - 凤心OS处理五行人格专项任务
  - 两者不互相嵌套，各自独立运行

Step 5：凤心OS Skills创建

文件结构：

~/.workbuddy/skills/凤心OS/
├── SKILL.md（主文件·核心定义）
└── references/
    ├── scene-router.md（场景路由矩阵·S0-S8）
    ├── fast-slow-thinking.md（快慢思考协调协议）
    └── wuxing-agents-protocol.md（五行分智能体协作协议）

SKILL.md必须包含的核心内容（6项缺一不可）：

1. 核心定位：发动机·心脏·神经中枢（1+5模式）
2. 快思考层：五维感知矩阵
   - 情绪信号（阴木愤怒/阴火急躁/阴土猜疑/阴金挑剔/阴水恐惧·各8分）
   - 行为信号（各4分）
   - 诉求类型（辅助·各2分）
3. 慢思考层：七步标准分析流程
   ① 一心状态定位 → ② 三界扫描 → ③ 五行生克诊断
   → ④ 九层定位 → ⑤ 转化方案 → ⑥ B=MAP落地 → ⑦ 输出报告
4. 路由决策规则：
   - 总分≥10 → 激活对应五行分智能体（精准慢思考）
   - 总分7-9 → 双行分析（主行+副行并行）
   - 总分5-6 → 全面诊断（五行总体分析）
   - 总分≤4  → 先追问1个精准问题，确认后路由
5. 龙心OS七模块整合表
6. 不触发场景（避免误触发）

Step 6：凤心OS全局Rule创建

路径：~/.workbuddy/rules/凤心OS自动触发.mdc
关键设置：alwaysApply: true
触发层级：
  第一层：上下文感知（情绪信号·行为信号）→ 最高优先级
  第二层：直接关键词（五行相关词汇）

Step 7：凤心OS MVP验证

测试场景：输入含情绪信号的对话
样例："我最近很烦躁，总跟同事吵架，他们就是不理解我"

期望结果：
  - 凤心OS感知到阴火信号（烦躁=急躁·8分）
  - 自动路由到火行分智能体
  - 输出火行三界诊断+拔阴取阳建议

通过标准：无需用户输入「凤心OS」等关键词，系统自动感知并分析
未通过：检查Rule文件，调整感知矩阵分值，确认alwaysApply:true

SKILL.md必须包含的5项核心内容：

1. 核心定位：顶层觉知接待台（超越五行的元状态）
2. 三步觉知流程：
   ① 叩问意图（4类标准问题）
   ② 感知握手包（场景×五行线索×情感温度×优先级）
   ③ 温暖分发（路由到对应场景）
3. 中庸原则：不过度热情·不冷漠机械·恰到好处
4. 与凤心OS的协作规则：
   - 单一场景→直接移交凤心OS
   - 多场景同时→L1主导整合
   - 情感温度极高→情感陪伴优先
5. 情感温度识别（6象限：愤怒/焦虑/热情/迷茫/平静/痛苦）

📋 附录C：常见问题与解决方案

问题 原因 解决方案
Skills在某些对话不触发 Rule不是alwaysApply 检查Rule文件，确认alwaysApply: true，Rules放在用户全局目录
凤心OS和龙心OS冲突 没有明确边界 龙心OS处理通用认知；凤心OS处理五行人格专项；两者不互相嵌套
L7知识调用不准确 知识标签不规范 统一标签体系，每篇都加五行标签+场景标签+L7根系标签
IMA无法执行 Windows权限问题 手动运行Python脚本，或直接使用IMA网页端
分智能体分析不准确 L7根系不够深 回到L7，补充该行人格的详细内容，特别是九层阶梯和转化技术
系统越来越复杂难维护 没有统一的文档索引 用📚 五行人格心理学总索引.md做统一入口，每次新增文档都更新索引
用户反馈不准确 缺乏反馈闭环 每次分析后收集用户打分，用于优化感知矩阵的分值权重
