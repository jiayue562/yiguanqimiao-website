## 📋 完整操作流程

### 场景 1: 主动保存（推荐）

**何时使用**: 你计划中断工作，下次继续

**操作步骤**:
1. 输入触发词：`保存上下文` 或 `生成快照`
2. （可选）补充说明：`我明天继续做这个`
3. AI 生成快照，告知存储位置
4. 确认快照内容（快速浏览）
5. 结束会话

**示例对话**:
```
你：保存上下文，这个研究报告还需要 2-3 次会话才能完成

AI: ## ✅ 上下文已保存（长周期项目模式）
    
    **快照文件**: `context-20260414-0930.md`
    **项目类型**: 长周期研究项目
    **当前进度**: 第 2/5 次会话
    
    ### 保存摘要
    - **项目**: AI 转型研究报告
    - **已完成**: 文献综述、15 个案例收集
    - **待办**: 继续收集案例 (目标 30 个)、搭建分析框架
    
    ### 下次恢复
    下次会话时输入"恢复任务"即可继续
```

### 场景 2: 自动提醒保存

**何时使用**: 会话即将超时，或对话轮数过多

**触发条件**:
- 对话轮数 >20 且任务未完成
- 会话超时前 30 分钟

**AI 主动提醒**:
```
AI: 检测到这是一个长周期任务，当前已对话 22 轮。
    
    建议保存上下文快照，以便下次会话快速恢复。
    是否需要我现在帮你生成快照？
    
    [是，保存上下文] [不，我快完成了]
```

### 场景 3: 恢复任务

**何时使用**: 新会话开始，继续之前的工作

**操作步骤**:
1. 输入触发词：`恢复任务` 或 `继续上次的项目`
2. AI 查找最新快照
3. AI 输出恢复摘要
4. 确认状态，开始继续工作

**示例对话**:
```
你：恢复任务

AI: ## 🔄 任务上下文已恢复
    
    **项目**: AI 转型研究报告
    **快照时间**: 2026-04-14 09:30
    **中断至今**: 16 小时
    
    ### 已完成事项（5 项）
    1. 文献综述（100%）
    2. 案例收集（15/30 个）
    3. ...
    
    ### 待办事项（4 项）
    - 🔴 高优先级：继续收集案例 (还需 15 个)
    - 🟡 中优先级：搭建分析框架
    ...
    
    ### 💡 下一步建议
    1. 先继续收集案例（目标 30 个，预计 1 小时）
    2. 然后搭建分析框架（预计 2 小时）
    
    ---
    *从快照恢复完成，可以开始继续工作了*
    
你：好的，那我们继续收集案例。目前已经收集了哪些行业的案例？
```

### 场景 4: 管理多个快照

**何时使用**: 有多个项目/多个版本的快照

**操作命令**:
```
列出所有快照：管理快照 list
删除过期快照：管理快照 delete context-20260410-0930.md
合并快照：管理快照 merge context-*.md
导出备份：管理快照 export zip
```

**示例**:
```
你：管理快照 list

AI: ## 📋 可用快照列表
    
    | 文件名 | 项目 | 时间 | 大小 |
    |--------|------|------|------|
    | context-20260414-0930.md | AI 转型研究 | 2 天前 | 3.2KB |
    | context-20260413-1545.md | Python 开发 | 3 天前 | 2.8KB |
    | context-20260410-0930.md | 市场分析 | 6 天前 | 4.1KB |
    
    共 3 个快照，总大小 10.1KB
    
    需要恢复哪个？输入文件名或"最新"
```

## 🎯 高级用法

### 技巧 1: 快照命名约定

**推荐格式**:
```
context-YYYYMMDD-HHMM-[项目缩写].md

示例:
context-20260414-0745-ai-research.md
context-20260414-0930-python-dev.md
```

**好处**:
- 时间排序清晰
- 项目识别快速
- 便于搜索过滤

### 技巧 2: 快照内容优化

**保存前检查清单**:
- [ ] 项目名称是否清晰？
- [ ] 已完成事项是否完整？
- [ ] 待办事项是否有优先级？
- [ ] 关键决策是否有依据？
- [ ] 相关文件路径是否正确？
- [ ] 下一步建议是否可执行？

**人工 Review 示例**:
```
AI: 快照已生成，请快速确认内容：
    
    ### 待确认事项
    - 待办事项中的"添加日志功能"优先级是否正确？
    - 相关文件路径是否需要更新？
    - 是否有遗漏的关键信息？
    
    [确认保存] [修改后再保存]
```

### 技巧 3: 多项目并行管理

**场景**: 同时进行多个项目，需要分别保存上下文

**策略**:
1. 每个项目独立快照
2. 快照文件名包含项目标识
3. 恢复时明确指定项目

**示例**:
```
你：保存上下文，这是 Python 开发项目的进度

AI: ✅ Python 开发项目上下文已保存
    快照文件：context-20260414-0745-python-dev.md

---

（切换到另一个项目）

你：保存上下文，这是 AI 研究报告的进度

AI: ✅ AI 研究报告上下文已保存
    快照文件：context-20260414-0930-ai-research.md

---

（恢复时）

你：恢复 Python 项目的任务

AI: 🔄 Python 开发项目上下文已恢复
    ...
```

### 技巧 4: 快照与长期记忆联动

**场景**: 重要决策需要同时保存到快照和长期记忆

**操作**:
```
你：保存上下文，并把关键决策也保存到长期记忆

AI: ✅ 上下文快照已保存
    快照文件：context-20260414-0930.md
    
    ✅ 关键决策已保存到 MEMORY.md:
    - 选择 pandas 而非 numpy（性能考虑）
    - 采用 pytest 作为测试框架
    - 项目架构采用模块化设计
    
    下次会话时：
    - 快照用于恢复具体任务状态
    - 长期记忆用于跨项目知识复用
```

## ⚠️ 常见问题与解决方案

### Q1: 快照文件找不到了怎么办？

**原因**: 存储路径变更、文件名记错

**解决方案**:
```
你：我找不到之前的快照了

AI: 正在搜索所有快照文件...
    
    找到以下快照:
    1. ~/.OpcCla 知识库/snapshots/context-20260414-0930.md
    2. ~/.OpcCla 知识库/snapshots/context-20260413-1545.md
    
    需要恢复哪个？或者我帮你加载最新的？
```

### Q2: 快照内容不完整怎么办？

**原因**: 保存时信息提取不完整

**解决方案**:
```
你：这个快照里缺少了 XXX 信息

AI: 明白，我来补充这部分信息。
    
    请告诉我:
    1. 缺少的具体信息是什么？
    2. 这部分信息的重要性（高/中/低）？
    
    我会更新快照并重新保存。
```

### Q3: 多个快照如何选择？

**原因**: 同一项目有多个版本的快照

**解决方案**:
```
你：这个项目有多个快照，应该用哪个？

AI: 检测到该项目有 3 个快照:
    
    | 版本 | 时间 | 进度 | 建议 |
    |------|------|------|------|
    | v1 | 2 天前 | 30% | 已过时 |
    | v2 | 1 天前 | 60% | 推荐 |
    | v3 | 3 小时前 | 80% | 最新 |
    
    建议恢复 v3（最新），除非你需要回退到之前的状态。
    需要我对比 v2 和 v3 的差异吗？
```

### Q4: 快照恢复后状态不一致怎么办？

**原因**: 快照保存后手动完成了部分工作

**解决方案**:
```
你：恢复后我发现有些待办已经完成了

AI: 明白，快照是上次保存时的状态。
    
    请告诉我:
    1. 哪些待办已经完成了？
    2. 是否有新的待办事项？
    
    我会更新当前状态，并可选地更新快照。
```

## 📊 最佳实践清单

### 保存时机
- [ ] 每次会话结束前（即使只是短暂中断）
- [ ] 完成一个重要阶段后
- [ ] 做出关键决策后
- [ ] 会话超时前 30 分钟

### 快照质量
- [ ] 项目名称清晰明确
- [ ] 已完成事项有进度百分比
- [ ] 待办事项有优先级标注
- [ ] 关键决策有依据说明
- [ ] 相关文件路径准确
- [ ] 下一步建议可执行

### 恢复流程
- [ ] 确认快照版本（最新 or 特定版本）
- [ ] 快速浏览恢复摘要
- [ ] 确认待办事项优先级
- [ ] 更新状态变化（如有）
- [ ] 开始继续工作

### 长期维护
- [ ] 每周清理过期快照（>30 天）
- [ ] 每月归档重要快照到长期记忆
- [ ] 定期备份快照目录
- [ ] 保持快照命名一致性

## 🎓 进阶：上下文工程思维

### 从"被动保存"到"主动管理"

**Level 1: 被动保存**
- 想起时才保存
- 恢复时手动查找
- 快照质量参差不齐

**Level 2: 主动管理**
- 每次会话结束自动保存
- 快照命名规范统一
- 定期清理和归档

**Level 3: 系统思维**
- 快照与长期记忆联动
- 多项目并行管理
- 快照内容持续优化

### 上下文工程的终极目标

```
从"每次重新开始解释" → "无缝接力继续工作"
     ↓                        ↓
  浪费时间和 token          流畅高效体验
  容易偏离主题              精准聚焦任务
  重复劳动                  持续积累
```
