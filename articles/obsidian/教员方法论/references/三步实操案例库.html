二、企业战略分析案例

案例背景

场景：某连锁餐饮企业是否进入华南市场

---

第一步：找（找到成熟的思维模型）

``` 匹配模型： ├── 战略分析：SWOT × → 可用 ├── 问题诊断：5Why × → 可用 ├── 方案设计：商业画布 × → 可用 └── 组织分析：组织能力九宫格 × → 可用

选定组合：SWOT + 商业画布 + 矛盾论

改装框架： SWOT → 加入"资源约束"维度 商业画布 → 加入"文化维度" ```

第二步：装（将模型改装成适配框架）

``` 改装三问：

问1：SWOT的核心结构是什么？ → 优势/劣势/机会/威胁 四象限

问2：我的具体问题需要调整哪些部分？ → 需要加入"资源约束"维度（资金/人才/供应链）

问3：如何让模型适配我的现实情况？ → 形成"战略四维+资源约束"框架 ```

第三步：填（填入具体现实内容）

| 四维 | 内容 | 资源约束 | |------|------|----------| | 优势（S） | 品牌影响力强、产品标准化程度高 | 西南团队成熟 | | 劣势（W） | 缺乏华南经验、供应链未覆盖 | 需新建配送体系 | | 机会（O） | 华南市场增速20%、消费升级 | 需2名区域负责人 | | 威胁（T） | 当地品牌强势、商业综合体入场费高 | 需150万启动资金 |

---

矛盾定位：

``` 根本矛盾：规模扩张 vs 组织能力 阶段矛盾：市场机会 vs 团队经验 主要矛盾：华南市场机会 vs 供应链缺失 矛盾主要方面：供应链缺失 ```

PDCA闭环：

``` P：Q2开2家试点店，验证华南模型 D：组建3人华南团队，建立1家核心供应商 C：Q3复盘（试点店坪效/客单价/复购率） A：根据结果决定Q4是否规模化 ```

三、个人职业发展案例

案例背景

场景：职场人面临是否跳槽的选择

---

第一步：找（找到成熟的思维模型）

``` 匹配模型： ├── 决策分析：决策矩阵 × → 核心可用 ├── 问题诊断：5Why × → 辅助可用 ├── 能力分析：能力冰山 × → 辅助可用 └── 价值分析：职业价值观 × → 核心可用

选定组合：决策矩阵 + 能力冰山 + 职业价值观

改装框架： 决策矩阵 → 加入"时间维度" 职业价值观 → 融入矛盾分析 ```

第二步：装（将模型改装成适配框架）

``` 改装三问：

问1：决策矩阵的核心结构是什么？ → 选项/利/弊/权重/得分

问2：我的具体问题需要调整哪些部分？ → 需要加入"时间维度"（短期/中期/长期影响）

问3：如何让模型适配我的现实情况？ → 形成"时间加权决策矩阵" ```

第三步：填（填入具体现实内容） 决策矩阵（时间加权）：

| 选项 | 短期利 | 中期利 | 长期利 | 短期弊 | 中期弊 | 长期弊 | 综合得分 | |------|--------|--------|--------|--------|--------|--------|----------| | 留在现公司 | +2 | +3 | +1 | -1 | -2 | -3 | +0 | | 跳槽A公司 | +4 | +3 | +2 | -3 | -1 | -1 | +4 | | 跳槽B公司 | +3 | +4 | +4 | -2 | -2 | 0 | +7 |

---

矛盾定位：

``` 根本矛盾：个人成长需求 vs 平台发展空间 阶段矛盾：短期收入 vs 长期发展 主要矛盾：晋升通道不畅 矛盾主要方面：晋升通道制度问题 ```

PDCA闭环：

``` P：先用内部沟通尝试解决，不行再跳槽 D：与直属领导坦诚沟通晋升诉求 C：1个月后评估领导反馈 A： ├── 沟通有效 → 内部转岗，留在公司 └── 沟通无效 → 启动跳槽B公司计划 ```

四、团队矛盾调解案例

案例背景

场景：销售部门与运营部门互相推诿，客户投诉率高

---

第一步：找（找到成熟的思维模型）

``` 匹配模型： ├── 矛盾分析：五行生克 × → 核心可用 ├── 沟通分析：非暴力沟通 × → 辅助可用 ├── 组织分析：利益相关方 × → 核心可用 └── 流程分析：RACI × → 辅助可用

选定组合：五行生克 + 非暴力沟通 + 利益相关方

改装框架： 五行生克 → 识别木克土（木=销售·刚直、土=运营·保守） 非暴力沟通 → 提供沟通话术模板 ```

第二步：装（将模型改装成适配框架））

``` 改装三问：

问1：五行生克的核心结构是什么？ → 木克土=抗上不服 vs 固执保守

问2：我的具体问题需要调整哪些部分？ → 需要具体化为"销售刚直 vs 运营保守"的冲突

问3：如何让模型适配我的现实情况？ → 形成"冲突诊断→沟通调解→机制固化"三步 ```

第三步：填（填入具体现实内容） 矛盾诊断：

``` 五行归类： ├── 销售部 → 木行人主导（刚直、目标导向） └── 运营部 → 土行人主导（保守、流程导向）

五行生克分析： 木克土 → 销售觉得运营死板，运营觉得销售乱来

主要矛盾：跨部门协作流程缺失 矛盾主要方面：流程定义不清（双方都有责任） ```

沟通调解（非暴力沟通话术）：

``` 对销售部： "当你们说'运营太死板'时（观察） 我感受到'困惑'（感受） 因为'我们不了解运营的具体限制'（需要） 我希望'能一起制定一个双方都认可的协作流程'（请求）"

对运营部： "当你们说'销售太乱来'时（观察） 我感受到'无奈'（感受） 因为'我们不了解销售的业务压力'（需要） 我希望'能提前了解销售需求，共同优化流程'（请求）" ```

PDCA闭环：

``` P：Q1制定《销售-运营协作SOP》 D：每周跨部门例会，每日信息同步 C：Q1末复盘（客户投诉率是否下降） A： ├── 投诉率下降>30% → SOP标准化，全公司推广 └── 效果不明显 → 深入分析，引入第三方调解 ```

五、项目问题定位案例

案例背景

场景：某软件开发项目延期3个月，预算超支50%

---

第一步：找（找到成熟的思维模型）

``` 匹配模型： ├── 问题诊断：鱼骨图 × → 核心可用 ├── 问题诊断：5Why × → 核心可用 ├── 风险管理：风险矩阵 × → 辅助可用 └── 项目管理：PDCA × → 辅助可用

选定组合：5Why + 鱼骨图 + 风险矩阵

改装框架： 5Why → 深入追问5层根因 鱼骨图 → 六类原因分析（人/机/料/法/环/测） ```

第二步：装（将模型改装成适配框架）

``` 改装三问：

问1：5Why的核心结构是什么？ → 连续追问5层"为什么"

问2：我的具体问题需要调整哪些部分？ → 需要结合鱼骨图的六类原因，避免遗漏

问3：如何让模型适配我的现实情况？ → 形成"5Why+鱼骨图"双层诊断框架 ```

第三步：填（填入具体现实内容） 5Why深层追问：

``` 问题：项目延期3个月

Why1：为什么延期？ → 需求变更频繁

Why2：为什么需求变更频繁？ → 需求确认流程缺失

Why3：为什么需求确认流程缺失？ → 产品经理与客户直接对接，无评审机制

Why4：为什么产品经理与客户直接对接？ → 缺乏项目边界定义，客户认为"改需求免费"

Why5：为什么缺乏项目边界定义？ → 合同条款不清晰，未明确变更流程

根本原因：合同管理缺失 ```

鱼骨图分析：

``` 项目失败 │ ┌───────┬───────┬───┴───┬───────┬───────┐ │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ 人员 机器 材料 方法 环境 测量 │ │ │ │ │ │ 沟通不足 工具老旧 第三方依赖 流程缺失 疫情影响 KPI不合理 ```

---

矛盾定位：

``` 根本矛盾：项目范围 vs 合同约束 阶段矛盾：快速交付 vs 质量保证 主要矛盾：需求管理失控 矛盾主要方面：需求管理流程缺失（可归因于甲方） ```

PDCA闭环：

``` P：Q2制定《需求变更管理流程》，重新签订补充协议 D： ├── 建立需求评审委员会 ├── 实施双周迭代机制 └── 每月客户汇报进度 C：Q2末评估（延期是否追回、成本是否控制） A： ├── 进度追回 → 流程标准化，其他项目复制 └── 仍有延期 → 重新评估项目范围，协商延期 ```
